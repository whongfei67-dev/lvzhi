# 律植项目发布 SOP（内测版）

> 适用范围：`apps/api`、`apps/web`、`apps/admin`，生产域名 `www.lvxzhi.com`

## 0. 固定动作（提需求 -> 改代码 -> 推送部署 -> 线上核验）

> 这是每次上线都执行的四步闭环。任何一步未完成，不进入下一步。

### 0.1 提需求（冻结范围）

- [ ] 用 1~3 句话写清楚需求目标（改什么、给谁用、验收标准）
- [ ] 明确影响范围：`API / Web / Admin / DB / Nginx / OSS`
- [ ] 明确是否需要数据迁移和回滚预案
- [ ] 明确上线后如何验证（具体页面、按钮、接口）

### 0.2 改代码（本地完成）

- [ ] 在功能分支或当前分支完成代码修改
- [ ] 至少执行一次本地检查（typecheck/lint/关键路径手测）
- [ ] 提交信息写清楚“为什么改”（不是只写“改了什么”）
- [ ] 推送到远端并确认 `git status` 干净

### 0.3 推送线上部署（ECS 执行）

- [ ] 登录 ECS，进入目录：`cd /opt/lvzhi`
- [ ] 先做磁盘预检：`./deploy/disk-preflight.sh --min-free-gb 8`
- [ ] 再执行更新：`./deploy/update-on-ecs.sh`
- [ ] 最后看状态：`docker compose ps`

### 0.4 线上核验（必须留痕）

- [ ] 核验容器健康：`api/web/admin/nginx/postgres` 均为 `healthy` 或稳定 `Up`
- [ ] 核验用户路径（页面级）：入口是否可见、点击是否生效、状态是否变化
- [ ] 核验后台路径（管理级）：数据总览/审核台是否读到最新数据
- [ ] 核验接口路径（服务级）：核心接口返回码正常（200/302）
- [ ] 记录结果（截图或文字）到“发布记录模板”

### 0.5 故障分流（固定判定）

- 若报 `502 Bad Gateway` 且刚重启不久：先等 30~60 秒后重查 `docker compose ps`
- 若报 `no space left on device`：先清理 Docker 空间，再重建容器
- 若按钮“看不到”：优先排查路由是否一致、登录态是否正确、是否命中“本人隐藏/禁用”逻辑

### 0.6 生产常用命令（固定）

```bash
cd /opt/lvzhi
./deploy/disk-preflight.sh --min-free-gb 8
./deploy/update-on-ecs.sh
docker compose ps
docker compose logs --tail=120 web admin api nginx
```

## 1. 发布节奏与顺序

- 固定顺序：`数据库迁移（如有） -> API -> Web -> Admin`
- 原因：先保证接口可用，再切前端页面，最后切后台
- 原则：每步发布后都做最小冒烟验证，不一次性全量切换

---

## 2. 发布前检查清单（必须全勾选）

### 2.1 环境与配置

- [ ] 生产环境变量已确认（DB、JWT、OSS、第三方 Key）
- [ ] `.env` 内无测试密钥/默认口令
- [ ] DNS 正常：`www.lvxzhi.com` 指向生产 ECS
- [ ] HTTPS 正常：`80 -> 443` 强制跳转已生效
- [ ] Nginx 路由确认：
  - [ ] `/` -> `apps/web`
  - [ ] `/admin` -> `apps/admin`
  - [ ] `/api` -> `apps/api`

### 2.2 质量与测试

- [ ] `apps/web` typecheck 通过
- [ ] `apps/admin` typecheck 通过（如有）
- [ ] `apps/api` 单测/关键用例通过（如有）
- [ ] 核心链路手测通过（登录、支付、审核、通知、导出）

### 2.3 数据与安全

- [ ] RDS 自动备份已开启（保留至少 7~14 天）
- [ ] 本次有迁移时，已做一次发布前快照
- [ ] RBAC 权限串测通过（用户/管理员/超管）
- [ ] OSS 权限最小化（只开必要读写）

### 2.4 监控与回滚

- [ ] 5xx 告警已开启（钉钉/短信）
- [ ] CPU、内存、磁盘、DB连接数告警已开启
- [ ] 回滚版本已确定（上一个稳定 tag / commit）
- [ ] 回滚流程可在 5~10 分钟内执行

---

## 3. 发布执行步骤（无数据库变更）

### 步骤 A：代码与构建

1. 合并代码到发布分支
2. 执行检查
   - `pnpm --filter web typecheck`
   - `pnpm --filter admin typecheck`（如有）
   - `pnpm --filter api test`（如有）
3. 构建产物
   - `pnpm --filter api build`
   - `pnpm --filter web build`
   - `pnpm --filter admin build`

### 步骤 B：部署

1. 部署 `apps/api`
2. 验证 API 健康检查（如 `/health`）
3. 部署 `apps/web`
4. 验证首页与关键用户路径
5. 部署 `apps/admin`
6. 验证后台登录与关键审批动作

### 步骤 C：观察

1. 发布后观察 10~30 分钟：
   - 错误率（5xx）
   - 平均响应时间
   - 登录失败率
2. 记录发布结果（时间、版本、异常情况）

---

## 4. 发布执行步骤（有数据库迁移）

1. 先做 RDS 快照/确认可恢复点
2. 执行 migration（遵循“先兼容再切换”）
3. 按顺序发布：`API -> Web -> Admin`
4. 抽查关键业务数据写入
5. 观察告警与错误日志

---

## 5. 冒烟测试清单（发布后立即执行）

### 5.1 用户端（Web）

- [ ] 首页、营销页打开正常
- [ ] 注册/登录/退出正常
- [ ] 搜索、详情页、发布流程正常
- [ ] 上传/下载、CSV 导出正常（如适用）

### 5.2 后台（Admin）

- [ ] 管理员登录正常
- [ ] 审核台、认证台、数据总览可访问
- [ ] 审核动作有审计日志
- [ ] 超管权限与普通管理员权限隔离正确

### 5.3 接口（API）

- [ ] 健康检查正常
- [ ] 关键接口响应正常（无大面积 4xx/5xx）
- [ ] 数据库连接池正常，无异常飙升

---

## 6. 紧急回滚 SOP

### 触发条件（任一满足即回滚）

- 登录失败率持续异常
- 核心支付/审核链路不可用
- 5xx 持续升高且无法快速止血

### 回滚顺序

1. 先回滚 `Web/Admin` 到上个稳定版本
2. 若问题在接口，回滚 `API`
3. 若问题来自迁移，执行回滚 SQL 或恢复快照

### 回滚后验证

- [ ] 登录/核心页面恢复
- [ ] 关键接口恢复
- [ ] 5xx 回落到正常范围
